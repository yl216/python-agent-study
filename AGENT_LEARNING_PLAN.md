# Python Agent 学习路线规划

本项目的目标是：从一个 10 行以内的简单交互程序开始，逐步理解并实现一个可以调用 DeepSeek API、能使用工具、能读取上下文、能拆解任务、能辅助学习和开发的 Python Agent。

学习方式采用“先能跑，再理解，再扩展”的节奏。每个阶段都要留下可运行代码、学习笔记和一次小复盘。

## 总体路线

| 阶段 | 主题 | 目标产物 |
| --- | --- | --- |
| 0 | 环境准备 | Python 项目骨架、虚拟环境、依赖管理 |
| 1 | 10 行以内交互程序 | 一个命令行问答小程序 |
| 2 | 接入 DeepSeek API | 一个可以调用模型并返回回答的脚本 |
| 3 | 对话记忆 | 支持多轮上下文的命令行聊天程序 |
| 4 | Prompt 设计 | 可切换角色、任务和输出格式的 Agent 原型 |
| 5 | 配置与密钥管理 | 使用 `.env` 管理 API Key 和模型参数 |
| 6 | 工具调用 | Agent 能调用本地 Python 函数完成任务 |
| 7 | 文件读写 | Agent 能读取、总结、改写本地文本文件 |
| 8 | 结构化输出 | Agent 输出 JSON，并由程序解析执行 |
| 9 | 简单任务规划 | Agent 能把复杂任务拆成步骤 |
| 10 | 小型项目 Agent | 一个面向学习或开发的完整 CLI Agent |

## 阶段 0：环境准备

目标：

- 安装并确认 Python 版本。
- 创建项目目录。
- 学会使用虚拟环境。
- 学会用 `pip` 安装依赖。
- 建立基础文件结构。

建议结构：

```text
study01/
  AGENT_LEARNING_PLAN.md
  README.md
  .env.example
  requirements.txt
  src/
    main.py
  notes/
    00_environment.md
```

需要理解：

- Python 解释器是什么。
- 虚拟环境为什么重要。
- 依赖包是什么。
- `.env` 文件为什么不能提交真实密钥。

验收标准：

- 能运行 `python --version`。
- 能创建并激活虚拟环境。
- 能运行一个 `print("hello agent")` 程序。

## 阶段 1：10 行以内的简单交互代码

目标：

写出最小版本的交互程序：用户输入一句话，程序返回固定回答。

示例方向：

```python
name = input("你叫什么名字？")
print(f"你好，{name}，我是你的第一个 Agent。")
```

需要理解：

- `input()` 和 `print()`。
- 字符串和变量。
- 程序的输入、处理、输出。
- 什么是“交互”。

扩展练习：

- 输入学习目标，程序返回鼓励语。
- 输入命令，例如 `help`、`exit`。
- 用 `while True` 实现持续对话。

验收标准：

- 代码不超过 10 行。
- 能正常接收用户输入。
- 能根据输入返回不同文本。

## 阶段 2：接入 DeepSeek API

目标：

让 Python 程序把用户输入发送给 DeepSeek 模型，并显示模型回复。

需要学习：

- HTTP API 的基本概念。
- 请求和响应。
- API Key。
- JSON 数据格式。
- DeepSeek 的模型名称、请求地址、消息格式。

建议依赖：

```text
openai
python-dotenv
```

说明：

DeepSeek API 通常兼容 OpenAI SDK 风格。正式编码前，需要以 DeepSeek 官方文档为准确认当前的 `base_url`、模型名和参数。

验收标准：

- API Key 不写死在代码里。
- 用户输入可以发送给 DeepSeek。
- 程序能打印模型回复。
- API 调用失败时能显示友好的错误信息。

## 阶段 3：多轮对话记忆

目标：

让 Agent 记住前几轮对话，而不是每次都像第一次聊天。

需要理解：

- `messages` 列表。
- `system`、`user`、`assistant` 三种消息角色。
- 上下文窗口。
- 为什么历史消息太多会变慢、变贵、变乱。

要实现：

- 启动后持续聊天。
- 每轮把用户输入加入历史。
- 每轮把模型回答加入历史。
- 输入 `exit` 退出。

验收标准：

- Agent 能理解前文。
- 能打印当前对话轮数。
- 能限制最多保留最近 N 轮对话。

## 阶段 4：Prompt 设计

目标：

让 Agent 不只是聊天，而是按照指定身份和任务方式工作。

需要理解：

- System Prompt。
- 角色设定。
- 输出格式约束。
- 少样本示例。
- Prompt 太长、太模糊、互相冲突时会发生什么。

练习方向：

- 学习老师 Agent。
- Python 代码解释 Agent。
- 错误排查 Agent。
- 文档总结 Agent。

验收标准：

- 能通过修改 system prompt 改变 Agent 行为。
- 能要求 Agent 使用固定格式输出。
- 能比较不同 prompt 的效果。

## 阶段 5：配置与密钥管理

目标：

把代码里的可变配置抽出来，让项目更像真实工程。

要实现：

- `.env.example` 保存示例配置。
- `.env` 保存本地真实配置。
- `settings.py` 统一读取配置。
- 支持配置模型名、温度、最大上下文轮数。

需要理解：

- 环境变量。
- 配置和代码分离。
- 为什么不能把 API Key 提交到仓库。

验收标准：

- 换模型不需要改主逻辑代码。
- 没有 API Key 时有清晰错误提示。
- 项目里有 `.env.example`。

## 阶段 6：工具调用

目标：

让 Agent 能使用本地函数，而不是只返回文字。

第一批工具：

- 计算器：执行简单数学计算。
- 时间工具：返回当前时间。
- 文件列表工具：列出当前目录文件。
- 笔记工具：把内容保存到 `notes/`。

需要理解：

- 函数就是工具。
- Agent 决定“要不要调用工具”。
- 程序负责真正执行工具。
- 模型输出和本地执行之间需要边界。

验收标准：

- 用户说“帮我保存一条笔记”时，程序能写入文件。
- 用户说“算一下”时，程序能调用计算工具。
- 工具执行失败时不会让整个程序崩溃。

## 阶段 7：文件读写能力

目标：

让 Agent 能围绕本地文件完成学习任务。

要实现：

- 读取一个 Markdown 文件。
- 总结文件内容。
- 提取待办事项。
- 根据用户要求追加学习笔记。

需要理解：

- 文件路径。
- 编码。
- 读写模式。
- Markdown 文档结构。

验收标准：

- Agent 能总结 `notes/` 里的指定文件。
- Agent 能把学习总结写回新文件。
- 文件不存在时有清晰提示。

## 阶段 8：结构化输出

目标：

让模型输出机器更容易理解的数据，例如 JSON。

需要理解：

- JSON 对象和数组。
- Python 的 `json.loads()`。
- 结构化输出比自然语言更适合驱动程序。
- 为什么需要校验模型输出。

练习方向：

- 让模型把用户意图解析为：

```json
{
  "intent": "save_note",
  "content": "今天学习了 input 和 print"
}
```

验收标准：

- 程序能解析模型输出的 JSON。
- JSON 格式错误时能要求模型重试或给出错误提示。
- 根据 `intent` 调用不同工具。

## 阶段 9：简单任务规划

目标：

让 Agent 能把一个复杂任务拆成多个步骤，并逐步执行。

示例任务：

```text
帮我学习 Python 函数，并生成一份练习题。
```

Agent 应拆成：

1. 解释函数概念。
2. 给出最小示例。
3. 生成练习题。
4. 等待用户作答。
5. 批改并给出下一步建议。

需要理解：

- 任务分解。
- 计划和执行分离。
- 每一步都要有可观察结果。
- Agent 不应该一次做太多不可验证的事。

验收标准：

- Agent 能输出步骤计划。
- 程序能逐步执行或逐步询问用户。
- 用户可以中途输入 `skip`、`retry`、`exit`。

## 阶段 10：小型项目 Agent

目标：

完成一个真正可用的命令行学习 Agent。

建议功能：

- 支持多轮聊天。
- 支持 DeepSeek API。
- 支持配置文件。
- 支持笔记保存。
- 支持读取学习资料。
- 支持生成练习题。
- 支持批改用户答案。
- 支持查看学习进度。

建议命令：

```text
/help       查看命令
/note       保存笔记
/review     复习最近笔记
/quiz       生成练习题
/plan       生成学习计划
/exit       退出
```

验收标准：

- 新用户能按照 README 跑起来。
- 常见错误有提示。
- 每个核心功能都有一个最小测试。
- 项目结构清晰，后续可以继续扩展。

## 每次学习的固定流程

每次学习都按下面顺序进行：

1. 明确今天目标。
2. 写一个最小可运行版本。
3. 运行并观察结果。
4. 解释每一行代码。
5. 做一个小改造。
6. 记录今天学到的概念。
7. 留一个下一次可以继续的问题。

- 询问用户名字。
- 询问用户今天想学什么。

限制：

- 不调用 API。
- 不超过 10 行。
- 必须能直接运行。
